# Deployment

How to get the backend online so a real APK works.

## Why an APK needs this at all

In development the mobile app finds the backend by reading the LAN IP from
Metro. A standalone build has no Metro, so it cannot discover anything — the
API URL has to be compiled into the build, and it has to be **HTTPS**:

- `android/app/src/debug/AndroidManifest.xml` sets `usesCleartextTraffic="true"`,
  but that is the **debug** manifest. Release builds have no such flag, and
  Android blocks plain `http://` without it.
- `src/lib/api.ts` now throws on a standalone build when `EXPO_PUBLIC_API_URL`
  is missing, instead of silently falling back to `localhost` — which on a
  phone means the phone itself.

## 1. Put the code in a Git repo

Render deploys from GitHub/GitLab. This project is not a repo yet:

```bash
cd c:/Users/User/Documents/announcement
git init
git add .
git commit -m "Initial commit"
git remote add origin https://github.com/<you>/<repo>.git
git push -u origin main
```

`backend/.gitignore` already excludes `.env`, `env/` and `uploads/`, so no
secrets or local images are pushed. **Check `git status` before the first push.**

## 2. The free stack, and why it is split across three services

Everything below is free, but two pieces cannot live on Render's free tier
without losing data:

| Piece | Where | Why not Render free |
|---|---|---|
| API | Render (free web service) | — |
| Admin console | Render (static site) | — |
| **Postgres** | **Neon** | Render's free database is deleted when its trial ends |
| **Event images** | **Cloudflare R2** | Free instances have no disk; files vanish on redeploy |

The trade-off you keep: a free Render service **sleeps after ~15 minutes
idle**, and the next request takes roughly 50 seconds to wake it. On the phone
that looks like the app hanging on launch. Nothing breaks, but it is the reason
paid instances exist.

### 2a. Database — Neon

1. neon.tech → new project → copy the connection string
2. It looks like
   `postgresql://user:pass@ep-xxx.aws.neon.tech/neondb?sslmode=require`
3. Keep `?sslmode=require` — Neon refuses plaintext connections

Nothing in the code changes; this is just a different `DATABASE_URL`.

### 2b. Image storage — Cloudflare R2

R2's free tier covers 10 GB with no egress charges. Uploads go to the bucket
instead of local disk as soon as all five `S3_*` variables are set — miss one
and the API silently falls back to the filesystem, so set them together.

1. Cloudflare dashboard → **R2** → create a bucket (e.g. `event-images`)
2. **Settings → Public access** → enable it, and copy the public URL
   (`https://pub-xxxxx.r2.dev`). Without this the app can read nothing back.
3. **Manage R2 API Tokens** → create a token with **Object Read & Write**
4. Note the values:

| Variable | Where it comes from |
|---|---|
| `S3_ENDPOINT_URL` | `https://<account-id>.r2.cloudflarestorage.com` |
| `S3_ACCESS_KEY_ID` | From the API token |
| `S3_SECRET_ACCESS_KEY` | From the API token — shown once |
| `S3_BUCKET` | Your bucket name |
| `S3_PUBLIC_URL` | The `pub-xxxxx.r2.dev` URL from step 2 |

Images stored this way get absolute URLs. Both clients already pass absolute
URLs through untouched, so no frontend change is needed.

### 2c. Deploy

`render.yaml` at the repo root describes both Render services. In Render:
**New → Blueprint**, pick the repo.

| Service | What |
|---|---|
| `announcement-api` | FastAPI on a free instance |
| `announcement-admin` | admin-web as a static site, with SPA rewrites |

Then fill in the values Render cannot know, all marked `sync: false` in the
blueprint:

On **`announcement-api`**:

- `DATABASE_URL` — the Neon string from 2a
- `CORS_ORIGINS` — the admin site URL, e.g. `https://announcement-admin.onrender.com`.
  The mobile app needs no entry; CORS applies to browsers, not native apps.
- The five `S3_*` values from 2b

On **`announcement-admin`**:

- `VITE_API_URL` — the API's full URL including `https://`, then trigger a
  rebuild. Vite inlines it at build time.

## Environment variables — the full picture

**No `.env` file is ever deployed.** `.env` is gitignored in every project, so
production values live in the host's dashboard instead. Your local `.env` files
stay on your machine and keep working for development, untouched.

### API service (Render dashboard → Environment)

| Variable | Set by | Value |
|---|---|---|
| `SECRET_KEY` | blueprint | Generated by Render, never leaves the dashboard |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | blueprint | `1440` (24h) |
| `MAX_UPLOAD_MB` | blueprint | `15` |
| **`DATABASE_URL`** | **you** | Neon connection string, keep `?sslmode=require` |
| **`CORS_ORIGINS`** | **you** | The admin site URL |
| **`S3_ENDPOINT_URL`** | **you** | `https://<account-id>.r2.cloudflarestorage.com` |
| **`S3_ACCESS_KEY_ID`** | **you** | R2 API token |
| **`S3_SECRET_ACCESS_KEY`** | **you** | R2 API token — shown once |
| **`S3_BUCKET`** | **you** | Bucket name |
| **`S3_PUBLIC_URL`** | **you** | `https://pub-xxxxx.r2.dev` |

The five `S3_*` values are all-or-nothing: with any one missing the API writes
to local disk instead, and on a free instance those files are gone by the next
deploy. `UPLOAD_DIR` is unused once object storage is on.

`SECRET_KEY` uses `generateValue: true`, so Render creates a strong random
value and nobody — including this setup — ever sees it. If you need to set one
by hand instead:

```bash
python -c "import secrets; print(secrets.token_urlsafe(64))"
```

Treat it like a password. **Changing it signs out every user**, because it is
the key every issued JWT was signed with.

### Admin site (Render dashboard → Environment)

| Variable | Value |
|---|---|
| **`VITE_API_URL`** | `https://announcement-api.onrender.com` |

Vite inlines this at build time, so changing it requires a rebuild, not just a
restart.

### Mobile app (`eas.json`, not `.env`)

| Variable | Where |
|---|---|
| **`EXPO_PUBLIC_API_URL`** | `eas.json` → `build.preview.env` and `build.production.env` |

This one lives in `eas.json` on purpose: `.env` is not uploaded to EAS Build,
so a value there would silently be missing from the APK. `eas.json` holds no
secret — it is a public URL, and anything shipped in an app binary is readable
by anyone who installs it. **Never put a real secret in `EXPO_PUBLIC_*`.**

### Local development is unaffected

`backend/.env` and `admin-web/.env` keep working exactly as they do now. The
mobile app still finds the backend from Metro's LAN IP with no `.env` at all.

### If you later want to pay for it

The one thing money buys here is the service not sleeping. Upgrading the API to
a paid instance removes the ~50 second cold start; nothing else has to change,
because Postgres and images already live outside Render.

Free-tier terms change — confirm current limits with Neon, Cloudflare and
Render rather than trusting this file.

### Migrations

`startCommand` runs `alembic upgrade head` before uvicorn starts. Keep it that
way — deploying new code against an old schema is what makes endpoints 500 with
`relation "..." does not exist` until someone migrates by hand.

## 3. Verify the deployment

```bash
API=https://announcement-api.onrender.com

curl $API/api/health                  # {"status":"ok"}
curl $API/api/settings                # public display settings
curl -I $API/uploads/<some-file>.png  # 200 + image/png
```

The upload check matters most: it is the exact path the mobile app builds image
URLs from, and a 404 there means every event card renders blank.

## 4. Create the first admin

`POST /api/auth/register` only ever creates `role=user`, so the first admin has
to be promoted directly. From the Render Postgres shell:

```sql
UPDATE users SET role = 'admin' WHERE email = 'you@example.com';
```

## 5. Build the APK

Set the real URL in `mobile-app/eas.json` — replace `REPLACE-ME` in **both**
the `preview` and `production` profiles:

```json
"env": { "EXPO_PUBLIC_API_URL": "https://announcement-api.onrender.com" }
```

Then:

```bash
cd mobile-app
npm install -g eas-cli
eas login
eas build:configure
eas build --platform android --profile preview     # installable APK
eas build --platform android --profile production  # AAB for Play Store
```

`preview` produces an APK you can sideload for testing. `production` produces
an app bundle, which is what the Play Store takes.

Because `EXPO_PUBLIC_*` values are inlined at build time, **changing the API
URL means rebuilding the app** — it cannot be reconfigured after install.

## Known limitations

- **The API sleeps.** A free Render service goes idle after ~15 minutes, and
  waking it takes roughly 50 seconds. On the phone that reads as the app
  hanging on launch. This is the main thing a paid instance fixes.
- **Old local images 404 after the switch.** Anything uploaded before R2 was
  configured is still recorded as `/uploads/...` in the database and no longer
  exists on disk. Re-upload those events' images, or leave them blank.
- **Nothing deletes from the bucket.** Replacing an event's image uploads a new
  object and abandons the old one. Harmless at this scale — 10 GB is a lot of
  event photos — but it grows forever.
- **`requirements.txt` includes `pytest` and `httpx`**, so production installs
  the test dependencies too. Harmless, just slower builds; split them out if
  that bothers you.
